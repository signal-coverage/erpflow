# HealthCore - Architecture.md

> Modular ERP Platform for Healthcare

Version: 0.1.0

---

# Vision

HealthCore is a SaaS platform for healthcare organizations built around a **Core + Plugin** architecture.

The Core manages generic healthcare operations.

Plugins implement specialty-specific workflows.

One platform should support:

- Individual professionals
- Clinics
- Medical centers
- Hospitals
- Health insurance providers

---

# Principles

- Core first
- Plugin driven
- Multi-tenant
- Cloud native
- Read-optimized (Firestore)
- Extensible without modifying the Core

---

# Core Responsibilities

The Core never contains specialty-specific logic.

Domains:

- Authentication
- Organizations
- Users
- Roles & Permissions (RBAC)
- Patients
- Professionals
- Scheduling
- Billing
- Notifications
- Reports
- Dashboard
- Audit Logs
- Plugin Engine
- Settings

# Plugin Responsibilities

Every specialty is an independent plugin.

Examples:

- Odontology
- Nutrition
- Psychology
- Laboratory
- Cardiology
- Dermatology

A plugin may register:

- Routes
- Navigation
- Widgets
- Permissions
- Firestore collections
- Events
- Reports
- Settings
- Components
- Services

# Golden Rules

1. The Core never contains specialty-specific logic.

2. Plugins never modify the Core.

3. Every document belongs to an Organization.

4. Patients belong to the Core.

5. Clinical information belongs to Plugins.

6. Plugins communicate through Events.

7. Navigation is generated dynamically.

8. Permissions are explicit.

9. Business logic lives in Services.

10. Components never access Firestore directly.

11. Every feature must work in a multi-tenant environment.

12. Prefer composition over inheritance.

13. Firestore is optimized for reads, not normalization.

14. Keep the Core stable. Add features through Plugins.

# A Recommendation Before Writing a Single Line of Code

After going through this entire exercise, I would change just one thing from the original plan: **I wouldn't start by building the Plugin Engine.**

Instead, I would develop the project in this order:

1. Project bootstrap (React, Firebase, Tailwind CSS, shadcn/ui)
2. Authentication system
3. Complete application layout (sidebar, header, navigation, theme)
4. Organizations + RBAC (Role-Based Access Control)
5. Patients
6. Professionals
7. Scheduling (Appointments & Calendar)
8. **Only then** implement the Plugin Engine

The reasoning is simple: by the time you build the Plugin Engine, you'll already have a fully functional Core to test it against.

This significantly reduces complexity and allows you to validate your MVP before investing time in a sophisticated plugin infrastructure.

In fact, with this approach, you could have your **first paying customer** using the system long before the second plugin even exists.

Build the platform that solves real problems first. Then make it extensible.
