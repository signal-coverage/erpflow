# Domain Model

Version: 0.1.0

---

# Purpose

This document defines the business language used throughout ERPFlow.

Every class, component, route, service, Firestore collection, API endpoint and plugin should use the terminology defined here.

This document is the source of truth for the domain.

---

# Core Philosophy

ERPFlow is composed of two domains:

- Core Domain
- Plugin Domains

The Core manages generic healthcare operations.

Plugins manage specialty-specific workflows.

# Organization

## Definition

An Organization represents a customer using ERPFlow.

It is the highest business entity.

Everything belongs to an Organization.

Examples

- Dental Clinic
- Nutrition Office
- Hospital
- Medical Center
- Health Insurance Provider

Relationships

Organization

↓

Users

Patients

Professionals

Appointments

Billing

Plugins

Settings

Nutritionist

Administrator

Owner

Manager

Relationships

Organization

↓

User

↓

# Professional

## Definition

A healthcare professional.

May or may not have a User account.

Examples

Dentist

Nutritionist

Psychologist

Cardiologist

Kinesiologist

Examples

A clinic may register a visiting doctor without providing system access.

---

# Patient

## Definition

A person receiving healthcare services.

The Patient is one of the most important entities in the platform.

The Core owns patient identity.

Plugins own clinical information.

Patient responsibilities

Identity

Contact Information

Insurance

Emergency Contacts

General Notes

Not included

Diagnosis

Treatments

Clinical History

Measurements

Those belong to plugins.

---

# Appointment

## Definition

A scheduled interaction between a Patient and a Professional.

Appointments belong to the Core.

Plugins extend appointments with clinical workflows.

Example

Appointment

↓

pluginId

↓

Nutrition

↓

Diet Evaluation

Another

Appointment

↓

pluginId

↓

Odontology

↓

Dental Cleaning

---

# Consultation

## Definition

The medical interaction that happens during an Appointment.

Unlike an Appointment,

Consultation belongs to a Plugin.

Examples

Dental Consultation

Nutrition Consultation

Psychology Session

Laboratory Analysis

The Core does not know what happens inside a Consultation.

# Ubiquitous Language

The following terms should always be used consistently.

| Preferred    | Avoid                                   |
| ------------ | --------------------------------------- |
| Organization | Company                                 |
| User         | Employee                                |
| Professional | Doctor (unless specialty-specific)      |
| Patient      | Client                                  |
| Appointment  | Booking                                 |
| Consultation | Visit                                   |
| Service      | Product                                 |
| Plugin       | Module (when referring to specialties)  |
| Module       | Plugin (when referring to Core domains) |
| Permission   | Access                                  |
| Role         | Profile                                 |

# Domain Rules

Rule 1

Every Patient belongs to one Organization.

Rule 2

Every Appointment belongs to one Patient.

Rule 3

Every Appointment references one Plugin.

Rule 4

Every Consultation belongs to a Plugin.

Rule 5

The Core never stores clinical information.

Rule 6

Plugins never duplicate Patient identity.

Rule 7

Permissions are always explicit.

Rule 8

Events describe completed business actions.

Rule 9

Widgets never contain business logic.

Rule 10

Plugins extend the platform.

They never modify the Core.

---

# Summary

ERPFlow has a small, stable business vocabulary.

The Core remains generic.

Medical knowledge lives inside Plugins.

Every architectural decision should reinforce this separation.

If a new concept cannot be clearly classified as either Core or Plugin, the design should be revisited before implementation.
